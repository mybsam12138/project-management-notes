# Kanban Board vs Scrum Board

## Quick Summary

A **Scrum board** mainly shows the work selected for the **current sprint**, while a **Kanban board** shows work flowing continuously through the system, usually focusing on **ready**, **in-progress**, and **done** work.

---

## Difference Table

| Aspect | Scrum Board | Kanban Board |
|---|---|---|
| Work model | Sprint-based | Continuous flow |
| Time structure | Fixed iteration (for example, 1–2 weeks) | No fixed sprint |
| Board scope | Shows tasks committed for the current sprint | Shows tasks in the workflow |
| Backlog handling | Product backlog is usually separate from the board | Backlog may be separate, partially shown, or integrated |
| Task selection | Tasks are selected at the start of the sprint | Tasks are pulled continuously when capacity is available |
| Change during execution | Usually limited during the sprint | Changes can happen anytime |
| Reset behavior | Usually reset or refreshed every sprint | Not reset; work keeps flowing |
| Main focus | Completing the sprint commitment | Improving flow efficiency |
| WIP limit | Optional, not always enforced | Commonly enforced and important |
| Planning style | More structured and time-boxed | Lighter and flow-based |
| Typical columns | To Do / In Progress / Testing / Done | Backlog / Ready / In Progress / Testing / Done |
| “Ready” meaning | Often implied during sprint planning | Usually an explicit status before development starts |
| Best for | Planned feature delivery | Support, operations, bug fixing, continuous work |

---

## Core Idea

### Scrum board
- We choose a **batch of tasks** for a sprint.
- The board mainly tracks **that batch only**.
- Main question: **Did we complete what we committed to in this sprint?**

### Kanban board
- Work is pulled **continuously**.
- The board tracks the **flow of work**.
- Main question: **How smoothly and quickly does work move through the system?**

---

## Easy Way to Remember

- **Scrum = time-boxed delivery**
- **Kanban = continuous flow**

---

## Practical Interpretation

If a team is building planned features with clear sprint goals, a **Scrum board** is usually a better fit.

If a team handles many changing tasks such as production issues, support tickets, bug fixes, and small requests, a **Kanban board** is usually more suitable.

Some teams combine both approaches. This is often called **Scrumban**.
