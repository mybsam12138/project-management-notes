# Handling Early Completion and Unfinished Work in Scrum

## 1. When a Developer Finishes Work Early

### ✅ Principle

Developers should **continue delivering value** and not stay idle.

---

### 🔹 Recommended Actions (Priority Order)

#### 1. Pull Next Priority Item

* Check **Product Backlog**
* Align with Product Owner before pulling new work

---

#### 2. Help Teammates

* Assist with blocked tasks
* Support testing, debugging, or integration

---

#### 3. Improve Quality

* Refactor code
* Add unit tests
* Improve performance

---

#### 4. Reduce Technical Debt

* Clean code
* Optimize architecture or database queries

---

#### 5. Do Investigation (Spike)

* Research upcoming features
* Clarify requirements or design

---

### ⚠️ Important Rule

* Do NOT start a large new feature if it cannot be finished within the Sprint

---

## 2. When a Developer Does NOT Finish Work

### ❌ Core Rule

* Work that is not finished does **NOT meet Definition of Done (DoD)**
* It is **NOT included in the Increment**

---

### 🔹 Handling Options

#### 1. Split the Story (Preferred)

* Deliver completed part ✅
* Return remaining part to backlog ❌

---

#### 2. If Cannot Split

* Mark the entire story as **NOT DONE**
* Return it to **Product Backlog**

---

### 🔁 Next Steps

* Product Owner re-prioritizes the item
* Team may select it again in next Sprint

---

### ⚠️ Important Rule

* Do NOT automatically carry unfinished work into next Sprint

---

## 3. Root Cause Analysis

After the Sprint, team should analyze:

* Poor estimation?
* Task too large?
* Blockers or dependencies?
* Unclear requirements?

---

### 🔧 Improvements

* Break stories into smaller tasks
* Improve backlog refinement
* Enhance team communication

---

## 4. Key Principles

* **“Done means Done”** → no partial credit
* **“Stop starting, start finishing”**
* **Continuous value delivery over individual completion**
